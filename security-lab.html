High Severity OWASP A3

🎯 XSS — Cross-Site Scripting

ثغرة تسمح للمهاجم بحقن كود JavaScript خبيث داخل صفحات الويب، يُنفَّذ في متصفح المستخدم الضحية.

📖

ما هي الثغرة؟

XSS تحدث عندما يقبل الموقع مدخلات من المستخدم ويعرضها مباشرةً في الصفحة دون تنقية أو ترميز. المهاجم يُدخل كود JS خبيث يُنفَّذ في متصفح الضحية بكامل صلاحيات الصفحة.

⚡

كيف تحدث؟

المستخدم يُرسل نص يحتوي على وسم <script>. الخادم يحفظه ويُعيد عرضه. المتصفح يُنفّذه كجزء من كود الصفحة. المهاجم يسرق cookies أو يُعيد توجيه المستخدم.

⚠️ هذه الأمثلة للأغراض التعليمية فقط. لا تستخدم هذه الكودات على مواقع حقيقية دون إذن صريح من المالك.
PHP — كود مُعرَّض للثغرة ❌ 
// المستخدم يكتب اسمه في نموذج
<?php
  $name = $_GET['name'];
  // ⛔ خطأ: طباعة المدخل مباشرة بدون تنقية
  echo "مرحباً، $name";
?>

// المهاجم يُرسل الرابط التالي:
https://example.com/greet.php?name=<script>alert('XSS!')</script>
// النتيجة: يظهر popup في متصفح أي زائر للرابط
PHP — كود آمن ✅ 
<?php
  $name = $_GET['name'];
  // ✅ استخدم htmlspecialchars لتحويل الأحرف الخطرة
  echo "مرحباً، " . htmlspecialchars($name, ENT_QUOTES, 'UTF-8');
?>

// الآن <script> تُعرض كنص عادي وليس كود
🛡️

كيف تحمي من XSS؟

  • استخدم htmlspecialchars() في PHP أو textContent في JS لعرض المدخلات دائماً كنص وليس HTML.
  • فعّل Content Security Policy (CSP) في ترويسات HTTP لمنع تنفيذ scripts خارجية.
  • استخدم مكتبات موثوقة لـ Template Rendering مثل Twig أو Blade التي تُنقّي تلقائياً.
  • لا تضع بيانات حساسة في cookies بدون HttpOnly flag لمنع JS من الوصول إليها.
  • اعتمد مبدأ Whitelist: اقبل فقط المدخلات المتوقعة، ارفض أي شيء غير ذلك.
Critical OWASP A1

💉 SQL Injection

ثغرة تُمكّن المهاجم من التلاعب في استعلامات قاعدة البيانات عن طريق حقن كود SQL خبيث في المدخلات.

📖

ما هي الثغرة؟

تحدث عندما يُدمج الكود مدخلات المستخدم مباشرةً في استعلام SQL. المهاجم يُغيّر منطق الاستعلام بالكامل، فيسرق البيانات، يتجاوز تسجيل الدخول، أو يحذف الجداول بأكملها.

💀

ما الذي يمكن فعله؟

سرقة أسماء المستخدمين وكلمات المرور، تجاوز صفحة تسجيل الدخول، قراءة ملفات الخادم، تعطيل قاعدة البيانات بأكملها، وأحياناً تنفيذ أوامر على نظام التشغيل.

⚠️ هذه الأمثلة تعليمية بحتة. اختبار الثغرات يجب أن يتم فقط على بيئات تجريبية خاصة بك.
PHP — كود مُعرَّض للثغرة ❌ 
<?php
$user = $_POST['username'];
$pass = $_POST['password'];

// ⛔ خطأ: دمج المدخلات مباشرة في الاستعلام
$query = "SELECT * FROM users
  WHERE username = '$user'
  AND password = '$pass'";

$result = mysqli_query($db, $query);

// المهاجم يُدخل في حقل username:
// admin' OR '1'='1
// الاستعلام يصبح صحيحاً دائماً → دخول بدون كلمة مرور!
?>
PHP — كود آمن باستخدام Prepared Statements ✅ 
<?php
// ✅ استخدم Prepared Statements دائماً
$stmt = $db->prepare(
  "SELECT * FROM users WHERE username = ? AND password = ?"
);

// ربط المدخلات بأمان — لا يُنفَّذ كـ SQL أبداً
$stmt->bind_param("ss", $user, $pass);
$stmt->execute();

// حتى لو أدخل المهاجم: admin' OR '1'='1
// يُعامَل كنص عادي وليس كـ SQL
?>
🛡️

كيف تحمي من SQL Injection؟

  • استخدم Prepared Statements / Parameterized Queries في كل استعلام بدون استثناء.
  • استخدم ORM موثوق مثل Eloquent أو Doctrine بدلاً من كتابة SQL يدوياً.
  • امنح قاعدة البيانات أدنى صلاحية ممكنة (Least Privilege) — لا تستخدم حساب root.
  • أخفِ رسائل الأخطاء عن المستخدم النهائي — سجّلها في الخادم فقط.
  • اعتمد WAF (Web Application Firewall) كطبقة دفاع إضافية.
Medium OWASP A8

🔄 CSRF — Cross-Site Request Forgery

ثغرة تخدع المستخدم المُسجَّل دخوله ليُنفَّذ طلبات غير مقصودة في موقع يثق به.

📖

ما هي الثغرة؟

المتصفح يُرسل cookies تلقائياً مع كل طلب للموقع. المهاجم يصنع صفحة خبيثة تُرسل طلبات للموقع الضحية. لأن المستخدم مُسجَّل دخوله، الموقع يقبل الطلب ظناً أنه شرعي.

🎭

مثال حقيقي

أنت مُسجَّل دخولك في البنك. تفتح رابطاً من بريد إلكتروني مشبوه. الرابط يُرسل طلب تحويل مالي للبنك بدون علمك. البنك يُنفَّذ الطلب لأن cookie الجلسة موجودة!

⚠️ المعلومات التالية للتعلم والفهم فقط. التطبيق على أنظمة بدون إذن جريمة يُعاقب عليها القانون.
HTML — صفحة خبيثة تُنفَّذ CSRF ❌ 
<!-- الضحية يفتح هذه الصفحة وهو مُسجَّل دخوله في البنك -->
<html>
<body>
  <!-- نموذج مخفي يُرسل طلب تحويل مالي -->
  <form id="attack"
        action="https://bank.com/transfer"
        method="POST">
    <input type="hidden" name="to"    value="attacker_account">
    <input type="hidden" name="amount" value="5000">
  </form>

  <!-- يُرسل النموذج تلقائياً عند فتح الصفحة -->
  <script>
    document.getElementById('attack').submit();
  </script>
</body>
</html>
PHP — الحماية بـ CSRF Token ✅ 
<?php
// 1. إنشاء token عشوائي وحفظه في الجلسة
session_start();
if (!isset($_SESSION['csrf_token'])) {
    $_SESSION['csrf_token'] = bin2hex(random_bytes(32));
}
?>

<!-- 2. تضمين الـ token في النموذج -->
<form method="POST" action="/transfer">
  <input type="hidden" name="csrf_token"
         value="<?= $_SESSION['csrf_token'] ?>">
  <!-- باقي حقول النموذج -->
</form>

<?php
// 3. التحقق من الـ token عند استقبال الطلب
if (!hash_equals($_SESSION['csrf_token'], $_POST['csrf_token'])) {
    die('طلب غير صالح — CSRF مكشوف!');
}
?>
🛡️

كيف تحمي من CSRF؟

  • استخدم CSRF Tokens: رمز عشوائي سري في كل نموذج، يُتحقق منه عند الإرسال.
  • اضبط SameSite=Strict أو SameSite=Lax على cookies الجلسة.
  • تحقق من Origin / Referer headers على الطلبات الحساسة.
  • استخدم Double Submit Cookie كطريقة بديلة موثوقة.
  • اطلب إعادة إدخال كلمة المرور للعمليات الحساسة كالتحويلات المالية.
High OWASP A1

🔓 IDOR — Insecure Direct Object Reference

ثغرة تسمح للمستخدم بالوصول لموارد لا يملك صلاحية الاطلاع عليها عبر تغيير معرّف مرئي في الطلب.

📖

ما هي الثغرة؟

حين يستخدم الموقع معرّفات قابلة للتخمين (1, 2, 3...) في الروابط أو API، ولا يتحقق من أن المستخدم يملك صلاحية الوصول، يستطيع المهاجم تغيير الأرقام للوصول لبيانات الآخرين.

🔍

كيف يكتشفها المهاجم؟

يُسجّل دخوله ويلاحظ الروابط: /profile?id=1042. يُغيّر الرقم إلى 1041 أو 1043. إذا رأى بيانات مستخدم آخر — الثغرة موجودة!

⚠️ الوصول لبيانات الآخرين دون إذن جريمة. هذا المثال للتعلم فقط في بيئة تجريبية خاصة.
PHP — كود مُعرَّض للثغرة ❌ 
<?php
// المستخدم يطلب: /invoice.php?id=1055
$invoiceId = $_GET['id'];

// ⛔ خطأ: جلب الفاتورة بدون التحقق من المالك
$stmt = $db->prepare(
  "SELECT * FROM invoices WHERE id = ?"
);
$stmt->bind_param("i", $invoiceId);
$stmt->execute();

// أي مستخدم يستطيع رؤية فواتير الآخرين
// بمجرد تغيير رقم id في الرابط!
?>
PHP — كود آمن مع التحقق من الصلاحية ✅ 
<?php
session_start();
$userId    = $_SESSION['user_id']; // المستخدم الحالي
$invoiceId = $_GET['id'];

// ✅ اربط الطلب دائماً بـ user_id الحالي
$stmt = $db->prepare(
  "SELECT * FROM invoices
   WHERE id = ? AND user_id = ?"
);
$stmt->bind_param("ii", $invoiceId, $userId);
$stmt->execute();
$result = $stmt->get_result();

if ($result->num_rows === 0) {
  // لا توجد فاتورة أو لا صلاحية — نفس الرد لإخفاء المعلومة
  http_response_code(404);
  die('الصفحة غير موجودة');
}
?>
🛡️

كيف تحمي من IDOR؟

  • تحقق دائماً من ملكية المورد: ربط user_id من الجلسة بكل استعلام بيانات.
  • استخدم UUID أو GUID بدلاً من أرقام متسلسلة — أصعب للتخمين.
  • طبّق Indirect Reference Maps: استخدم معرّفات داخلية غير مرتبطة بـ DB مباشرة.
  • اعتمد مبدأ Deny by Default: امنع كل وصول ثم افتح ما هو مسموح فقط.
  • راجع كل نقطة API بـ اختبارات Authorization وتأكد أن كل endpoint محمي.
